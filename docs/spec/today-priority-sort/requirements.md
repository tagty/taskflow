## 概要

「判断の滞留をなくす」思想のもと、`/today` ビューを「今日何から着手するか」を即判断できる画面に強化する。
priority 順ソートと見積時間合計表示・過負荷警告を追加する。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: Issue・ユーザヒアリングを参考にした確実な要件
- 🟡 **黄信号**: 既存実装から妥当な推測による要件

### 通常要件

- REQ-001: システムは今日のタスクを priority 昇順（1が最優先）で表示しなければならない 🔵 *Issue #17・ヒアリングより*
- REQ-002: システムは priority が null のタスクを priority あり のタスクの後ろに表示しなければならない 🔵 *ヒアリングより*
- REQ-003: システムは今日のタスクの estimate_minutes 合計を時間変換して表示しなければならない 🔵 *Issue #17より*
- REQ-004: システムは合計 estimate_minutes が 480分（8h）を超えた場合に視覚的な警告を表示しなければならない 🔵 *ヒアリングより*

### 条件付き要件

- REQ-101: estimate_minutes が設定されているタスクが1件以上ある場合、システムは合計時間を表示しなければならない 🔵 *ヒアリングより*
- REQ-102: すべてのタスクの estimate_minutes が null の場合、システムは合計時間を表示しなくてよい 🟡 *既存 null 表示パターンから推測*

### 制約要件

- REQ-401: ソートはサーバー側（Server Component）で行わなければならない 🔵 *既存アーキテクチャより*
- REQ-402: 閾値は 480分（8h）固定とする 🔵 *ヒアリングより*

## 非機能要件

### パフォーマンス
- NFR-001: ソートは既存クエリ結果をメモリ内で並び替えるだけで、追加クエリは不要 🟡 *既存実装から推測*

### ユーザビリティ
- NFR-201: 過負荷警告は滞留タスク警告と視覚的に区別できること 🟡 *既存 UI パターンから推測*
- NFR-202: すべての新規 UI 要素に `dark:` クラスを付けること 🔵 *開発ルールより*

## Edgeケース

- EDGE-001: タスクが0件の場合、合計・警告は表示しない 🟡 *既存 null 表示から推測*
- EDGE-002: すべてのタスクが estimate_minutes=null の場合、合計は表示しない 🟡 *上記と同様*
- EDGE-003: priority が同じ値のタスクが複数ある場合は created_at 順を維持する 🟡 *安定ソートの慣例から推測*
