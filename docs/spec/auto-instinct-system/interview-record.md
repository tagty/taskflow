# auto-instinct-system ヒアリング記録

**作成日**: 2026-03-23
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

ECC の instinct システムを参考に、Claude Code 操作の自動記録・分析機能を taskflow プロジェクトに導入するにあたり、既存の `diary-on-stop.sh` / `diary-sync` システムとの差分・設計方針を明確化した。

## 質問と回答

### Q1: 記録対象パターン

**カテゴリ**: 未定義部分詳細化
**背景**: ECC は多様なパターンを記録するが、このプロジェクトで価値があるものを絞る必要があった

**回答**: ツール使用パターン・コマンドパターン・エラー→修正パターン（3種）

**信頼性への影響**:
- REQ-001, REQ-011 の信頼性が 🔴 → 🔵 に向上

---

### Q2: instinct ファイルの保存先

**カテゴリ**: 設計確認
**背景**: `.claude/` 配下への追加か、learnings.md への統合かの選択

**回答**: `.claude/instincts/`（専用ディレクトリ）

**信頼性への影響**:
- REQ-012 の信頼性が 🔴 → 🔵 に向上

---

### Q3: 自動分析のタイミング

**カテゴリ**: 設計確認
**背景**: PostToolUse は頻繁すぎる可能性。既存の Stop hook パターンと合わせるか確認

**回答**: Stop hook 時（diary-on-stop.sh と同タイミング）

**信頼性への影響**:
- REQ-010 の信頼性が 🟡 → 🔵 に向上

---

### Q4: 信頼度管理方式

**カテゴリ**: 設計確認
**背景**: ECC はAI評価方式を採用。発生回数ベースの方がシンプルで制御しやすい

**回答**: 発生回数ベース（1回=low、複数回=medium→high）

**信頼性への影響**:
- REQ-020〜022 の信頼性が 🔴 → 🔵 に向上

---

### Q5: diary-sync との連携方式

**カテゴリ**: 統合設計
**背景**: instinct を独立したサブシステムにするか、既存の diary-sync フローに統合するか

**回答**: instinct → diary-sync の入力に含める（既存フローを拡張）

**信頼性への影響**:
- REQ-030〜032 の信頼性が 🟡 → 🔵 に向上

---

## ヒアリング結果サマリー

### 確認できた事項
- 記録対象: ツール使用・コマンド・エラー→修正の3パターン
- 保存場所: `.claude/instincts/`
- 分析タイミング: Stop hook
- 信頼度: 発生回数ベース
- diary-sync 統合: instincts/ を追加入力として取り込む

### 残課題
- instinct の「同一性判定」ロジックの詳細（AI 意味的類似度で判断するが閾値は実装時に調整）
- `session-log.jsonl` の累積方式（同日複数セッション時の扱い）
